# ADR-0004: 派生计算发生在构建时,事实源只存解析后的静态值

- 状态:已采纳(2026-07-05)
- 关联:ADR-0001、ADR-0002

## 背景

成熟的 token 体系大量使用派生关系:间距按基准值倍增(`spacing-medium = 2 × spacing-base`)、状态色按公式推导(disabled = 主色降饱和)。旧系统(Sass / 表格)把公式写在事实源里。公式应该住在哪一层?

## 备选方案

1. **运行时计算**:CSS `calc()` / `color-mix()` 等。iOS、Android、Figma 不认识 CSS 函数,跨平台断裂;最终值不可审计。
2. **事实源内嵌公式**:DTCG 标准没有表达式语法,任何消费方都需要自带公式解释器,违背标准化初衷。
3. **构建时计算**:公式住在生成脚本(即一种摄入适配器,ADR-0002),事实源存执行后的静态值。

## 决策

采用**方案 3**。行业先例:Material Design 3(种子色 → HCT 算法 → 全套色调)、Adobe Leonardo(对比度驱动生成色阶)、Radix Colors——算法都在生成端,发布物全是静态值。

关键收益:公式改动后重新生成,git diff 精确列出受影响的每个 token,**公式的影响面变成可 review 的清单**。

补充决策:未来自有色彩公式迁移时,色彩计算从 HSL/Sass 函数升级到 **OKLCH**(感知均匀色彩空间),使"降饱和 58%"这类操作在不同色相上视觉效果一致,减少人工例外。

## 后果

- 事实源可收敛为"少量设计决策 + 公式"的配置文件,由生成脚本展开为全量 DTCG(规划为 Phase 1.5)
- 真正需要运行时动态性的场景(如流体排版 `clamp()`)作为例外单独评估
- Figma 同步(Phase 2)推送的是解析后静态值,与其能力边界(Variables 无表达式)天然匹配
