# ADR-0007: 设计提案落在 overrides 层——生成式契约与人工变更的共存

- 状态:已采纳(2026-07-10)
- 关联:ADR-0002(契约是适配器的生成物)、ADR-0005(Figma 是提案通道)、
  ADR-0006(插件路线)

## 背景

提案方向(Figma → git PR)上线后的**第一个真实提案 PR 被 CI 打回**。原因:
drift check(ADR-0002)会重跑摄入适配器再生契约并比对,而适配器从上游
(Spectrum)取值,把提案手改的 `color/blue/600` 覆盖回原值,diff 不为零。

这不是误报,而是两个各自正确的机制正面冲突:

- drift check 主张:**契约是生成物,不许手改**
- 提案通道主张:**设计师的改动就是对契约的手改**

任何提案 PR 都过不了 CI——洞在设计层面,不在实现层面。

## 备选方案

1. 放松 CI:提案分支豁免 drift check——门禁一旦有豁免通道,"契约无漂移"的
   承诺对所有分支都不再可信
2. 提案改写适配器源码里的映射表——要求设计师的提案以改 JS 的形式表达,门槛
   与角色错位
3. **overrides 层**:新增 `packages/tokens/overrides.json` 作为适配器的**输入**,
   记录"有意偏离上游"的值;适配器生成后应用它;提案 PR 同时提交**意图**
   (overrides)与**效果**(再生后的契约文件)
4. CI 机器人在提案分支上自动重跑适配器并追加 commit——PR 被机器改写,作者
   看到的不再是自己提交的内容

## 决策

采用 **overrides 层**(方案 3)。契约的生成等式从"契约 = f(上游)"升级为
"契约 = f(上游) + 已采纳的偏离",drift check 验证的正是这个等式。

设计原则:**刻意的偏离必须有显式身份。**它既不是上游的值,也不是随手的编辑,
而是一条可 review、可追溯、可单独撤销的记录。overrides.json 同时成为"我们与
上游全部分歧"的一页式清单。

## 后果

- 插件 Propose 流程提交两类文件;二者由 CI 强制自洽,伪造效果而不留意图
  (或反之)都会被 drift check 拒绝
- override 目标在上游消失时(stale override)适配器立即报错——偏离清单
  不允许静默腐烂
- 未来换上游(ADR-0002 的承诺)时,overrides.json 就是迁移检查单:每一条
  偏离都要重新决策"保留还是回归"
- 首例:`color/blue/600 → #f6fd72`(PR #2,2026-07-10,Figma 提案全链路
  首次闭环)
