# ADR-0008: MCP 服务器作为契约的第三个消费端——只读查询面

- 状态:已采纳(2026-07-16)
- 关联:ADR-0001(DTCG 契约)、ADR-0006(Figma 插件路线)、ADR-0007(overrides 层)

## 背景

仓库立项时的四阶段规划里有"MCP 服务器"一环(仓库名 `mcpmvp` 即是承诺),但
在 token 管线、Figma 双向同步、版本治理逐步落地的过程中,这一环从工作地图上
无声滑落——没有被做,也没有被正式砍掉。盘点原始目标时发现了这个缺口,需要
一个显式决定。

同时,AI 辅助开发已是设计基建的现实工作面:AI 编码助手写 UI 代码时对设计系统
一无所知,只能靠猜色值、猜变量名,产出与契约脱节的硬编码样式。传统解法是文档
站(人读),但 AI 消费端需要的是**协议化的查询接口**(机器读)。

## 备选方案

1. 正式砍掉:聚焦传统 DSE 职责,更快进入面试准备——但放弃了作品集里差异化
   最强、且仓库名已经承诺的一块
2. 文档站/Storybook:对人友好,但 AI 工具无法结构化调用,没有解决硬编码问题
3. **只读 MCP 服务器**:把 git 契约暴露成三个查询工具(search/resolve/suggest),
   AI 助手按角色取 token,而不是猜值
4. 读写 MCP 服务器:再加"修改 token"工具——但变更已有专门通道(Figma 插件
   → overrides → PR,ADR-0006/0007),再开一条写路径会绕过 review 门禁

## 决策

采用**只读 MCP 服务器**(方案 3),`packages/mcp-server/`,stdio 传输。

设计原则:**git 契约是唯一事实源,消费端只增不改。**MCP 是继平台产物
(CSS/TS/JSON)和 Figma 插件之后的第三个消费端;三者读同一份 DTCG 源文件,
所以答案永远一致。写路径维持唯一入口(提案 PR),门禁不被稀释。

工具面刻意最小:

- `search_tokens`:按名字/类型/层级检索,结果带可直接使用的 CSS 变量名
- `resolve_token`:展示 usage → semantic → primitive 完整别名链(分主题,
  暗色刻度重锚在链上直接可见)
- `suggest_token`:hex 反查最接近的 token,精确命中优先、同值时 usage 层
  优先——把"设计稿取色/代码里的硬编码"引导回契约

## 后果

- 服务器读取与 build.js 相同的主题源文件集,契约变更零同步成本;但 build.js
  的源集合若调整,`lib.mjs` 的 THEME_SOURCES 需同步(两处清单,已知耦合)
- 色彩相近度用 RGB 距离,对"最接近"的判断在感知上不精确;升级到 OKLCH 距离
  是已知的改进方向(与被搁置的 Phase 1.5 同一知识域)
- 只读决定意味着 AI 助手无法直接提交 token 变更提案;若未来需要,应复用
  overrides → PR 通道而不是绕过它
